# 方法論（グレーボックスDASTのフロー）

このプラグインは**LLM支援型のグレーボックスDAST**を行います。対象のソース（開いているリポジトリ）
を読み、それを手がかりに、OWASP ZAP とブラウザで稼働中のアプリを動的に診断します。純粋な
ブラックボックスではなく、人間のペンテスターの代替でもありません。

## 役割分担

- **LLM（Claude）** — ソースを読み、診断対象マップを構築し、カバレッジのギャップを分類する。
  工程6では、対象固有のペイロードを組み立てて能動的に探索・確認する。対象は2種類：(1) ロジック／
  認可／IDOR／業務フローの欠陥、(2) XSS・SQLi・テンプレート注入・SSRF などの注入系。ソースから
  文脈（テンプレートエンジン、DB、反射位置）を読んでペイロードを作り、ZAPが到達できない箇所
  （認証後・多段・JS依存・隠しパラメータ・JSON/GraphQL本文）へ送り、ZAPのアラートの確認や誤検知の
  切り分けも行う。加えて証拠の突き合わせ、事実と推測の分離、レポート作成を担う。
- **OWASP ZAP** — プロキシ＋Spider＋Passive Scan＋（ゲート付き）Active Scan＋アラートエンジン。
  通信の記録役であり、脆弱性スキャナ。
- **Playwright／ブラウザ** — クローラーが到達できない画面（ログイン、JS描画、複数ステップ、権限
  別）へ到達する。ZAP Proxy経由でZAPが通信を記録する。

## 工程の全体像

0. 実行条件・安全ゲート（`safety-policy.md`）。
1. ソースコード解析（`source-analysis.md`）。
2. 診断対象マップ → `target-map.md`（テンプレート `templates/target-map.example.md`）。
3. ZAP初期探索：Context＋スコープ、Spider、Passive、任意でAjax、収集
   （`zap-integration.md`）。
4. カバレッジ比較＋Playwright補完 → `coverage-analysis.md`（`scenario-testing.md`）。
5. Active Scan — ゲート付き・確認優先（`safety-policy.md`）。
6. シナリオベース診断 → `scenarios.md`（`scenario-testing.md`）。
7. 結果整理＋レポート → `findings.md`、`report.md`（`report-format.md`）。

## 診断対象マップ — 各エントリの必須項目

- URL / APIエンドポイント＋HTTPメソッド
- 認証要否＋必要な権限/ロール
- 入力箇所＋想定データ形式
- セキュリティ上の注目点／想定脆弱性
- 診断優先度（例：高/中/低＋理由）
- 根拠：ソース由来か推測か；未確認・オープンな疑問点

優先度の目安：認証/認可面、入力の多いエンドポイント、管理者機能、ファイルアップロード、金銭/PIIに
触れるものを最優先とする。

## チェックポイントと部分実行

- 工程ごとにサマリを出し、次へ進む前に確認する（工程5の前は必須）。
- `--only <step>` は1工程のみ、`--from <step>` は指定工程から再開。いずれの場合も工程0の安全
  ゲートが先に必ず走る。
- 各工程の成果物は `reports/dast/<run-id>/` 配下の個別ファイルとし、人間が1工程ずつ見直せる
  ようにする。
