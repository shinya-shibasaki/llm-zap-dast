# 工程1 — ソースコード解析

Claude Codeで現在開いているリポジトリ（診断対象のソース）を解析します。目的は、診断対象マップの
構築とZAP自動スキャンが見逃すシナリオ設計に足るだけ、攻撃面を理解することです。

## 抽出するもの

- **フレームワークと起動方法**：言語、Webフレームワーク、アプリの起動方法（開発サーバのコマンド、
  ポート、Docker、環境）。判明すれば実際の `base_url`/ポートも記録。
- **ルートとエンドポイント**：画面URL、APIエンドポイント、HTTPメソッド、path/query/bodyパラメータ、
  content type。ルーター、コントローラ、アノテーション、フレームワーク規約で定義されるルートを含む。
- **フォームとアップロード**：フォーム項目、CSRFトークンの扱い、ファイルアップロードのエンドポイントと
  型/サイズ制限。
- **認証／認可／セッション**：ログインフロー、セッション/Cookie/JWTの扱い、ロール/権限チェック、
  認可がどこで（あるいはどこで実施されずに）行われているか。
- **管理者機能・機微**：管理者専用機能、破壊的エンドポイント、外部通信（Webhook、メール、
  サードパーティAPI）、DBアクセスのパターン、シークレットの利用。
- **セキュリティ上重要なロジック**：入力検証、出力エンコード、SSRFの起きうる取得処理、
  デシリアライズ、テンプレート描画、リダイレクト、レート制限。

## ルール

- **各項目の根拠を明記する。** 各項目を `source`（コードで確認）か `inferred`（推測/仮定）で
  ラベル付けする。両者を決して混ぜない — レポートはこの分離に依存する。
- 推測より、ルート定義・設定・フレームワークのエントリポイントを読むことを優先する。
- **未確認の疑問点**（例：「認証ミドルウェアはあるが適用範囲が不明」）はメモに残し、断定しない。
- この工程ではアプリの起動やエクスプロイトを行わない — リポジトリを読むだけの静的解析。

## 着手の手がかり（スタックに応じて調整）

- JS/TS：`package.json` の scripts、`routes*`、`app.*`、`server.*`、Express/Koa/Next の
  ルートファイル、`pages/`・`app/` の API ルート。
- Python：`manage.py`、`urls.py`、`app.py`、FastAPI ルーター、Flask blueprint。
- Ruby：`config/routes.rb`。PHP：フレームワークのルートファイル。Java：コントローラの
  アノテーション。
- 設定/環境：`.env.example`、`docker-compose*`、ポートとホストのフレームワーク設定。

この工程の成果は工程2（診断対象マップ）へ渡ります。生メモは保持し、構造化したマップを
`target-map.md` に書きます。
