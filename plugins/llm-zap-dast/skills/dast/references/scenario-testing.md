# 工程4・6 — カバレッジ補完とシナリオ診断

## 工程4 — カバレッジ分析＋Playwright補完

1. ソース抽出URL（工程1）と、ZAPが実際に到達したURL（工程3）を**比較**する。未到達を次のいずれ
   かに分類：
   - ログイン必須／JavaScript操作必須／特定の画面遷移が必要／特定のデータが必要／管理者権限が
     必要／URLは存在するが未使用／APIが画面から直接呼ばれない／クローラーでは到達困難。
   `reports/dast/<run-id>/coverage-analysis.md` に保存。
2. Playwright（または利用可能なブラウザ操作）で、未到達画面へ**ZAP Proxy経由で**遷移し、ZAPに
   記録させる（`zap-integration.md` 参照）。ブラウザ操作が無い → スキップ（fail-soft）、理由を記録。

許可される操作：ログイン、メニュー操作、フォーム送信、JS生成画面、複数ステップ遷移、権限別画面
確認。

**自動実行してはならない操作**（工程4・6の両方）：データ一括削除、ユーザー削除、パスワード変更、
外部メール送信、課金、外部サービス登録、本番データ変更、その他復旧不能な操作。

## 工程6 — シナリオベース診断（能動探索）

**これは本プラグインの中核であり、「LLMを本気で使う」工程です。** ZAPの自動スキャンは汎用の
注入を広く機械的に撃ちますが、(1) 認証後・多段・JS依存・隠しパラメータ・JSON/GraphQL本文などの
**到達できない注入点**、(2) ソース文脈（テンプレートエンジン、DB、反射位置）に**合わせた
ペイロード**、(3) ロジック/認可/IDOR/業務フローの**推論**、が苦手です。ここでは
**LLM（あなた）自身が、ソース解析とZAP履歴から仮説を立て、対象ごとに狙いを定めたペイロード／
リクエストを組み立てて送信し、応答を観察して調整・再送し、脆弱性を確認または否定します。**
**対象はロジック系だけでなく、XSS・SQLi・テンプレート注入・SSRF などの注入系も含み、いずれも
ZAPを補完・強化します**（注入系がZAP専任という意味ではありません。下記のクラス別手引きを参照）。

この工程は `scan.scenario_tests: true`（既定）で**常に実行**し、**ZAP Active Scan（工程5）とは
独立**です。Active Scan が無効でも、LLMの能動探索はここで行われます。

### 安全レールと能動探索は別物

安全レールは「**何を壊してはいけないか／どこを触ってはいけないか**」を縛るものであって、
「LLMにペイロードを作らせて探索させるか」を縛るものではありません。**安全レールを守ったまま、
能動的に探索してください。** 具体的には：

- **スコープ**：`allowed_hosts` のホストのみ。`exclude.paths` を厳守。スコープ外へは1リクエストも
  送らない。
- **非破壊・検出止まり**：脆弱性は**最小の証拠で「存在する」ことを示すところまで**。悪用（データ
  奪取、権限奪取の実行、破壊）まではしない。良性の一意マーカーや読み取り優先で確認する。
- **可用性を守る**：DoS相当・大量リクエスト・重いtime-based探索はしない。レート確認等は**上限を
  明示した小さなバースト**に留める。
- **状態変更の線引き**：確認に**不可逆な状態変更や実害**が必要な場合は**自動実行しない**。最も
  安全な部分確認（例：認可チェックの有無だけを読み取りで確認）に留め、残りは**要人間**として記録。
- 削除・一括削除・ユーザー削除・パスワード変更・外部メール送信・課金・外部登録・本番データ変更・
  復旧不能操作は行わない。

### 能動探索ループ（各高優先度ターゲットに対して）

診断対象マップの優先度順に、次を回す：

1. **仮説**：ソース（認可の位置、入力の扱い等）＋ZAP履歴から「何がどう壊れ得るか」を立てる。
2. **プローブ作成**：そのエンドポイント固有のパラメータ／認証／content-type に合わせ、最小・非破壊の
   ペイロード／リクエストを**自分で組み立てる**（汎用の使い回しではなく文脈に合わせる）。
3. **送信**：curl / Playwright / ZAP のリクエスト送信APIで送る（**ZAP Proxy経由**で履歴に残す）。
   ベースライン（正常系）も取り、差分で判断できるようにする。
4. **観察**：ステータス・本文・ヘッダ・タイミング・反射・差分を読む。
5. **調整・再送**：結果に応じてペイロードを練り直し再送する（**ここがLLMの価値**。1発で諦めない）。
6. **結論**：確認／否定／要人間に分け、証拠（マスク済み）を残す。

### クラス別・安全プローブ手引き（検出レベル）

いずれも allowed_hosts 限定・非破壊・検出止まり。状態変更/悪用が要るものは要人間へ。

- **IDOR / オブジェクト認可**：2つの低権限テストアカウントで識別子を入れ替え、**読み取りGET**で
  他人のリソースが取れるか。まず read-only。
- **垂直権限昇格 / 機能認可**：管理者ルート/アクションを**低権限セッション/トークンで再送**し、
  拒否されるか。状態変更を伴う管理操作は読み取り版で確認、無ければ要人間。
- **認証回避 / セッション**：Cookie/JWT の改ざん（`alg:none`、署名なし、期限切れ、別人のもの）、
  無認証リクエスト、保護ルートへの強制ブラウズ。
- **CSRF**：トークンの有無/検証を確認。トークンを外した/改ざんしたリクエストが受理されるかを
  **良性・非破壊なアクション**で確認。破壊的アクションでしか確認できないなら要人間。
- **反射/格納 XSS**：**良性の一意マーカー**（例：`zdast<乱数>`）や無害な破壊確認プローブ
  （`"><svg data-zdast=1>` 等、実行はさせない）を入れ、**反射の有無とエスケープ文脈**を見る。
  実際に情報を盗む/実行する完成ペイロードは撃たない。
- **SQLi / NoSQLi**：**boolean差分**（`' OR '1'='1` と `' AND '1'='2` の応答差）や単一引用符の
  エラー誘発など**検出のみ**。stacked query、`UPDATE/DELETE/DROP`、重いtime-based（DoS相当）は
  しない。読み取りエンドポイント優先。
- **コマンド/テンプレート注入**：`${7*7}` / `{{7*7}}` → 49 のような**良性の評価確認**で検出。
  OSに影響する破壊的コマンドは撃たない。
- **オープンリダイレクト**：リダイレクト先を**良性の制御マーカーURL**にし、30x/Location を確認
  （実際に有害先へは飛ばさない）。
- **SSRF**：取得系パラメータを**良性のループバック/制御マーカー**に向けて反応を見る。実在の内部
  インフラを破壊的に叩かない。
- **Mass Assignment**：書き込み系に想定外フィールド（`role`、`is_admin` 等）を足して受理されるか。
  実権限が上がるならテストデータで行うか要人間。
- **HTTPメソッド改ざん**：`PUT/DELETE/PATCH/OPTIONS` 等を試す。破壊的 `DELETE` を実リソースへ
  自動発行しない（テストリソース or 要人間）。
- **隠し/未文書パラメータ**：ソースにあるがUIに出ないパラメータをファズし、挙動変化を見る。
- **レート制限**：**上限を明示した小さなバースト**で制限の有無を確認（DoSにしない）。

### 各シナリオに記録する

- シナリオID、対象機能、想定される脆弱性、根拠（なぜ疑うか）
- 前提条件、操作手順、**組み立てたペイロード/リクエスト**（機微はマスク）、**試した反復**
- 期待される安全な挙動 vs 脆弱だった場合の挙動
- 実行可否、実行結果、証拠、追加確認事項（要人間の残作業を含む）

### その他の制約

- 証拠は保存前にマスクする（`redaction.md` 参照）。
- 確認済み（再現した）と、疑い/要人間確認を明確に分ける。根拠のない断定はしない。

`reports/dast/<run-id>/scenarios.md` に保存（テンプレート
`templates/scenario-list.example.md`）。レポートへ進む前にチェックポイント。
