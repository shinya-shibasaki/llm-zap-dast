# 工程4・6 — カバレッジ補完とシナリオ診断

## 工程4 — カバレッジ分析＋Playwright補完

1. ソース抽出URL（工程1）と、ZAPが実際に到達したURL（工程3）を**比較**する。未到達を次のいずれ
   かに分類：
   - ログイン必須／JavaScript操作必須／特定の画面遷移が必要／特定のデータが必要／管理者権限が
     必要／URLは存在するが未使用／APIが画面から直接呼ばれない／クローラーでは到達困難。
   `reports/dast/<run-id>/coverage-analysis.md` に保存。
2. Playwright（または利用可能なブラウザ操作）で、未到達画面へ**ZAP Proxy経由で**遷移し、ZAPに
   記録させる（`zap-integration.md` 参照）。ブラウザ操作が無い → スキップ（fail-soft）、理由を記録。

許可される操作：ログイン、メニュー操作、フォーム送信、JS生成画面、複数ステップ遷移、権限別画面
確認。

**自動実行してはならない操作**（工程4・6の両方）：データ一括削除、ユーザー削除、パスワード変更、
外部メール送信、課金、外部サービス登録、本番データ変更、その他復旧不能な操作。

## 工程6 — シナリオベース診断（能動探索）

この工程では、LLMがソース解析（工程1）とZAP履歴（工程3）から仮説を立て、対象ごとにペイロード／
リクエストを組み立てて送信し、応答を観察して調整・再送し、脆弱性を確認または否定する。工程1〜4の
結果を入力とし、結果は工程7のレポートに渡す。

ZAP Active Scan（工程5）は汎用の注入を広く自動で撃つ。本工程はそれと独立に動き、次を担う。

1. ZAPが到達できない注入点（認証後・多段・JS依存・隠しパラメータ・JSON/GraphQL本文）への送信。
2. ソース文脈（テンプレートエンジン、DB、反射位置）に合わせたペイロードの作成。
3. ロジック・認可・IDOR・業務フローの推論。

対象はロジック系だけでなく、XSS・SQLi・テンプレート注入・SSRF などの注入系も含む。
`scan.scenario_tests: true`（既定）のとき実行する。Active Scan が無効でもこの工程は実行する。

### 安全制約

安全制約は「送信してよい範囲」と「行ってはならない操作」を定める。制約の内側であれば、ペイロードを
能動的に組み立てて送信してよい。

- スコープ：`allowed_hosts` のホストのみに送信する。`exclude.paths` を除外する。スコープ外へは
  送信しない。
- 検出止まり：脆弱性は最小の証拠で存在を示すところまでとし、悪用（データ奪取、権限奪取の実行、
  破壊）は行わない。良性の一意マーカーや読み取りを優先する。
- 可用性：DoS相当・大量リクエスト・重いtime-based探索は行わない。レート確認は上限を決めた小さな
  バーストに限る。
- 状態変更：確認に不可逆な状態変更や実害が必要な場合は自動実行しない。認可チェックの有無を読み取り
  で確認するなど安全な部分確認に留め、残りを要人間として記録する。
- 次の操作は行わない：削除・一括削除・ユーザー削除・パスワード変更・外部メール送信・課金・外部
  登録・本番データ変更・その他復旧不能な操作。

### 能動探索ループ

診断対象マップの優先度順に、各ターゲットへ次を適用する。

1. 仮説：ソース（認可の位置、入力の扱い等）とZAP履歴から、想定される欠陥とその根拠を挙げる。
2. プローブ作成：エンドポイント固有のパラメータ・認証・content-type に合わせ、最小・非破壊の
   ペイロード／リクエストを組み立てる。汎用の使い回しではなく文脈に合わせる。
3. 送信：curl / Playwright / ZAP のリクエスト送信APIで送る（ZAP Proxy経由で履歴に残す）。正常系の
   ベースラインも取得し、差分で判断できるようにする。
4. 観察：ステータス・本文・ヘッダ・タイミング・反射・差分を確認する。
5. 調整・再送：応答に応じてペイロードを修正し、必要なだけ再送する。
6. 結論：確認／否定／要人間に分け、証拠（マスク済み）を記録する。

### クラス別プローブ（検出レベル）

いずれも allowed_hosts 限定・非破壊・検出止まりとする。状態変更や悪用が必要なものは要人間とする。

- IDOR / オブジェクト認可：2つの低権限テストアカウントで識別子を入れ替え、読み取りGETで他人の
  リソースが取得できるか確認する。
- 垂直権限昇格 / 機能認可：管理者ルート／アクションを低権限セッション／トークンで再送し、拒否
  されるか確認する。状態変更を伴う操作は読み取り版で確認し、無ければ要人間とする。
- 認証回避 / セッション：Cookie/JWT の改ざん（`alg:none`、署名なし、期限切れ、別人のもの）、
  無認証リクエスト、保護ルートへの強制ブラウズを試す。
- CSRF：トークンの有無と検証を確認する。トークンを外した／改ざんしたリクエストが受理されるかを、
  良性・非破壊なアクションで確認する。破壊的アクションでしか確認できない場合は要人間とする。
- 反射/格納 XSS：良性の一意マーカー（例：`zdast<乱数>`）や無害な確認プローブ
  （`"><svg data-zdast=1>` 等、実行させない）を入力し、反射の有無とエスケープ文脈を確認する。
  情報を盗む／実行する完成ペイロードは送らない。
- SQLi / NoSQLi：boolean差分（`' OR '1'='1` と `' AND '1'='2` の応答差）や単一引用符による
  エラー誘発で検出する。stacked query、`UPDATE/DELETE/DROP`、重いtime-based（DoS相当）は行わ
  ない。読み取りエンドポイントを優先する。
- コマンド/テンプレート注入：`${7*7}` / `{{7*7}}` → 49 のような良性の評価確認で検出する。OSに
  影響する破壊的コマンドは送らない。
- オープンリダイレクト：リダイレクト先を良性の制御マーカーURLにし、30x/Location を確認する。
  実際に有害先へ遷移させない。
- SSRF：取得系パラメータを良性のループバック／制御マーカーに向けて反応を確認する。実在の内部
  インフラを破壊的に叩かない。
- Mass Assignment：書き込み系に想定外フィールド（`role`、`is_admin` 等）を加えて受理されるか
  確認する。実権限が上がる場合はテストデータで行うか要人間とする。
- HTTPメソッド改ざん：`PUT/DELETE/PATCH/OPTIONS` 等を試す。破壊的 `DELETE` を実リソースへ自動
  発行しない（テストリソース、または要人間）。
- 隠し/未文書パラメータ：ソースに存在しUIに現れないパラメータを送り、挙動の変化を確認する。
- レート制限：上限を決めた小さなバーストで制限の有無を確認する。DoSにしない。

### 各シナリオに記録する

- シナリオID、対象機能、想定される脆弱性、根拠
- 前提条件、操作手順、組み立てたペイロード／リクエスト（機微はマスク）、試した反復
- 期待される安全な挙動と脆弱時の挙動
- 実行可否、実行結果、証拠、追加確認事項（要人間の残作業を含む）

### その他の制約

- 証拠は保存前にマスクする（`redaction.md` 参照）。
- 確認済み（再現した）と、疑い／要人間確認を分ける。根拠のない断定はしない。

`reports/dast/<run-id>/scenarios.md` に保存（テンプレート
`templates/scenario-list.example.md`）。レポートへ進む前にチェックポイント。
