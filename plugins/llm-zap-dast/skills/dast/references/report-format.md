# 工程7 — 結果整理とレポート形式

ZAPアラート＋シナリオ結果を `findings.md` に分析し、次に最終 `report.md` を生成する
（テンプレート `templates/report.example.md`）。

## 証拠の分離（必須）

各検出を2軸で分類し、ラベルを付ける：

- **事実の出所**：ツール検出の事実／HTTPで確認できた事実／ソースで確認できた事実／LLMの推測。
- **確度**：再現できた／再現できなかった／誤検知の可能性／人間の確認が必要。

根拠のない断定をしない。LLMの推測はそのようにラベル付けし、確認済みの脆弱性として提示しない。

## report.md — 必須セクション（15）

1. 診断概要
2. 対象情報（URL、ホスト、スコープ、除外パス）
3. 実行日時
4. 使用したツール（ZAPバージョン、Playwright、スクリプト）
5. 実行した診断工程 — **スキップした工程とその理由を含む**
6. 探索範囲（カバレッジ）
7. 未到達範囲
8. 検出結果一覧
9. 再現確認結果
10. 証拠（マスク済み）
11. リスク説明
12. 修正案
13. 未確認事項
14. 診断上の制約（グレーボックス、v1は認証なし、キーなし＝ローカル限定 など）
15. 免責事項

## 免責事項の内容（セクション15）

次を明記する：これはLLM支援型のグレーボックスDASTであり、完全な手動ペンテストの代替ではない；
結果には誤検知/検知漏れが含まれ得る；許可されたホストのみを診断した；検出結果は対応前に人間の
確認を要する；本ツールは破壊的な検証を行わない。

## 併せて出力する

- `execution-summary.json` — 機械可読：run-id、実行 vs スキップした工程＋理由、タイムスタンプ/
  所要時間、ツールの利用可否、Active Scanを実行したか、マスク状況。
- レポート中の証拠はマスク済みであること（`redaction.md` 参照）。
